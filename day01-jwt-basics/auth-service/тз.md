CRUD целей:


Описание
Реализовать полный CRUD для целей пользователя. Каждая цель принадлежит конкретному пользователю - доступ к чужим целям запрещён. При удалении цели каскадно удаляются все её задачи и результаты.

POST /goals - тело запроса
{ "title": "Выучить TypeScript", "description": "...", "priority": "high",
"start_date": "2025-06-01", "deadline": "2025-09-01", "daily_time_minutes": 60 }
POST /goals - ответ 201
{ "data": { "id": "uuid", "title": "...", "status": "in_progress", "progress": 0, ... } }
PATCH /goals/:id - частичное обновление
{ "title": "Новое название", "priority": "medium" }  // только изменяемые поля
GET /goals - ответ 200
{ "data": [ { ...goal }, ... ], "total": 5 }
Бизнес-правила
При создании status = in_progress, progress = 0 - устанавливаются сервером.
updated_at обновляется автоматически при каждом PATCH.
deadline должен быть >= start_date, если оба переданы.
Пользователь видит только свои цели (фильтр по user_id из JWT).
При DELETE чужой цели - 403 Forbidden, несуществующей - 404 Not Found.
Критерии готовности:
POST создаёт цель, возвращает 201 с полным объектом.
GET /goals возвращает только цели текущего пользователя.
GET /goals/:id возвращает 404 для несуществующей цели.
PATCH обновляет только переданные поля, остальные не трогает.
DELETE удаляет цель и все связанные tasks и results.
Запрос к чужой цели возвращает 403.

Фильтрация и сортировка целей


Описание:
Расширить GET /goals поддержкой query-параметров для фильтрации по статусу и сортировки. Пользователь может выбрать один или несколько статусов одновременно. Изменений в схеме БД не требуется.
Query-параметры
Параметр
Тип
Пример
Описание
status
string
status=in_progress,frozen
Один или несколько статусов через запятую
sort
string
sort=priority
Поле сортировки: priority | created_at | deadline
order
string
order=asc
Направление: asc | desc. По умолчанию desc

Допустимые значения status:
in_progress - цели в работе
completed - завершённые цели
frozen - замороженные цели
archived - архив целей
Можно передавать несколько через запятую: ?status=completed,archived
Примеры запросов:
GET /api/v1/goals?status=in_progress
GET /api/v1/goals?status=completed,archived&sort=deadline&order=asc
GET /api/v1/goals?sort=priority&order=desc
Ответ 200
{ "data": [ ...goals ], "total": 3 }
Бизнес-правила:
Если status не передан - вернуть все цели пользователя.
Если sort не передан - сортировать по created_at DESC.
Неизвестный статус в параметре - вернуть 400 с описанием ошибки.
Фильтрация и сортировка выполняются на уровне SQL-запроса (не в памяти).
Критерии готовности:
status=in_progress возвращает только цели с этим статусом.
status=completed,archived возвращает цели с обоими статусами.
sort=priority сортирует корректно (high > medium > low или через числовое поле).
order=asc / desc работает для всех полей сортировки.
Неверный статус -> 400 с понятным сообщением.
Без параметров возвращаются все цели пользователя.

PATCH обновляет только переданные поля.
DELETE удаляет задачу, возвращает 204.
Запрос к задаче чужой цели ->403.
Несуществующая цель ->404.
Валидация: пустой title, неверный priority, progress вне 0–100 ->400.

AI-планирование


Описание
Реализовать endpoint, который принимает данные цели, формирует промпт и отправляет запрос к внешнему LLM (OpenAI или аналог). В ответ возвращает список предлагаемых задач. Пользователь просматривает список на FE и подтверждает - задачи сохраняются через стандартный POST /goals/:goalId/tasks.
Ключ API хранится только на сервере в переменных окружения. Во FE не передаётся.
Endpoint
Метод
URL
Auth
Описание
POST
/api/v1/ai/plan
✓ JWT
Сгенерировать план задач для цели

Тело запроса
{ "goal_id": "uuid" }
Сервер сам загружает цель из БД по goal_id и проверяет права доступа.
Ответ 200
{ "data": {
"tasks": [
{ "title": "Прочитать главу 1", "description": "...",
"estimated_minutes": 60, "priority": "high" },
{ "title": "Выполнить упражнения", "description": "...",
"estimated_minutes": 90, "priority": "medium" }
]
}
}
Структура промпта к LLM
Промпт собирается из данных цели. Пример системного сообщения:
Ты планировщик задач. Пользователь хочет достичь цели.
Сгенерируй список конкретных задач для достижения этой цели.
Верни ТОЛЬКО JSON-массив объектов без пояснений.
Каждый объект: { title, description, estimated_minutes, priority }
User-сообщение передаёт название, описание, дедлайн и доступное время в день из объекта Goal.
Бизнес-правила
Проверить, что goal_id принадлежит текущему пользователю - иначе 403.
AI_API_KEY читается из переменных окружения, не хардкодится.
Если LLM вернул невалидный JSON - вернуть 502 Bad Gateway с сообщением об ошибке.
Таймаут запроса к LLM - 30 секунд. При превышении - 504 Gateway Timeout.
Задачи из ответа НЕ сохраняются автоматически - только возвращаются пользователю.
Переменные окружения
Переменная
Описание
AI_API_KEY
Ключ API (OpenAI или другого провайдера)
AI_MODEL
Модель, напр. gpt-4o-mini
AI_TIMEOUT_MS
Таймаут в мс, по умолчанию 30000

Критерии готовности (DoD)
POST /ai/plan возвращает массив tasks с нужными полями.
Ключ API нигде не попадает в ответы и логи.
Чужая цель ->403. Несуществующая ->404.
Невалидный JSON от LLM ->502.
Таймаут LLM ->504.
При отключённом AI_API_KEY endpoint возвращает понятную ошибку конфигурации.
