# План реализации GoalWays Backend — 2 дня

## Контекст

Проект GoalWays — платформа для постановки целей с геймификацией. Frontend (React Native) готов как UI-мокап. Backend на Spring Boot (Java 21) имеет только базовую JWT-авторизацию (register/login). Нужно:
1. Реализовать ТЗ1 — регистрация с подтверждением email
2. Привести эндпоинты в соответствие с архитектурой (`/api/auth/*`, стандартный формат ответов, refresh token)
3. Реализовать Goals CRUD + Stages

## ДЕНЬ 1: ТЗ1 (Email Verification) + Auth Architecture (~5.5 ч)

### Шаг 1.1 — Стандартный формат ответа API (30 мин)
**Создать:**
- `dto/ApiResponse.java` — generic-обёртка `{"status": "success/error", "data/error": {...}}`
- `exception/GlobalExceptionHandler.java` — `@RestControllerAdvice` для единообразных ошибок

### Шаг 1.2 — Миграция: email, status, confirmation tokens (30 мин)
**Создать:** `db/migration/V2__add_email_verification.sql`
- `ALTER TABLE users ADD COLUMN email VARCHAR(255) UNIQUE`
- `ALTER TABLE users ADD COLUMN status VARCHAR(20) DEFAULT 'NOT_CONFIRMED'`
- `CREATE TABLE confirmation_tokens` (id, token, user_id, expires_at, created_at, used)

### Шаг 1.3 — Обновить модель User + ConfirmationToken entity (30 мин)
**Изменить:** `model/User.java` — добавить `email`, `status`
**Создать:**
- `model/ConfirmationToken.java`
- `repository/ConfirmationTokenRepository.java` (`findByToken`, `findTopByUserOrderByCreatedAtDesc`)

### Шаг 1.4 — BCrypt cost 12 + обновить регистрацию (30 мин)
**Изменить:** `service/AuthService.java`
- `new BCryptPasswordEncoder(12)` (ТЗ1 требует cost ≥ 12)
- В `register`: принимать email, ставить status = `NOT_CONFIRMED`, генерировать confirmation token (UUID, expiry 24ч)

**Изменить:** `dto/RegisterRequest.java` — добавить поле `email`

### Шаг 1.5 — Email-сервис (45 мин)
**Изменить:** `pom.xml` — добавить `spring-boot-starter-mail`
**Изменить:** `application.yml` — SMTP-конфиг (Gmail или консольный fallback для dev)
**Создать:** `service/EmailService.java` — метод `sendConfirmationEmail(to, token)` с ссылкой подтверждения

### Шаг 1.6 — Эндпоинты confirm + resend (45 мин)
**Изменить:** `controller/AuthController.java`
- Сменить `@RequestMapping` на `/api/auth`
- `GET /confirm?token=...` — подтверждение email (проверка токена, активация аккаунта)
- `POST /resend-confirmation` — повторная отправка (проверка cooldown 60 сек, генерация нового токена)

**Изменить:** `service/AuthService.java` — методы `confirmEmail(token)`, `resendConfirmation(email)`
**Изменить:** `repository/UserRepository.java` — добавить `findByEmail`
**Изменить:** `config/SecurityConfig.java` — добавить новые пути в `permitAll`

### Шаг 1.7 — Формат ответов register/login (30 мин)
**Изменить:** `controller/AuthController.java` — обернуть ответы в `ApiResponse`
- Register → 201: `{user_id, username, message}`
- Login → 200: `{access_token, refresh_token, token_type, expires_in}`

**Создать:** `dto/RegisterResponse.java`
**Изменить:** `dto/TokenResponse.java` — добавить `refresh_token`, `expires_in`

### Шаг 1.8 — Refresh Token (45 мин)
**Создать:** `db/migration/V3__add_refresh_tokens.sql` (таблица refresh_tokens)
**Создать:** `model/RefreshToken.java`, `repository/RefreshTokenRepository.java`
**Изменить:** `service/AuthService.java` — генерация refresh token при login
**Изменить:** `controller/AuthController.java` — `POST /refresh`

### Шаг 1.9 — Cleanup + E2E тест (30 мин)
**Изменить:** `AuthApplication.java` — убрать тестовый `CommandLineRunner`
**Тест:** Полный цикл: register → confirm → login → refresh → resend

## ДЕНЬ 2: Goals CRUD + Stages (~4.5 ч)

### Шаг 2.1 — Миграция: goals + stages (30 мин)
**Создать:** `db/migration/V4__create_goals_and_stages.sql`
- `goals`: id, user_id, title, description, status, created_at, updated_at
- `stages`: id, goal_id, title, is_completed, sort_order, created_at, completed_at

### Шаг 2.2 — Entity + Repository (30 мин)
**Создать:** `model/Goal.java`, `model/Stage.java`
**Создать:** `repository/GoalRepository.java` (`findByUserId` с пагинацией, `findByIdAndUserId`)
**Создать:** `repository/StageRepository.java` (`findByIdAndGoalId`)

### Шаг 2.3 — DTO для Goals/Stages (30 мин)
**Создать:** `dto/CreateGoalRequest.java`, `dto/GoalResponse.java`, `dto/GoalListResponse.java`, `dto/CreateStageRequest.java`, `dto/StageResponse.java`

### Шаг 2.4 — GoalService (45 мин)
**Создать:** `service/GoalService.java`
- `createGoal(userId, title, description)` → Goal
- `getGoals(userId, page, size)` → Page<Goal>
- `getGoalWithStages(userId, goalId)` → Goal (с проверкой владельца)

### Шаг 2.5 — StageService (30 мин)
**Создать:** `service/StageService.java`
- `addStage(userId, goalId, title, sortOrder)` → Stage
- `completeStage(userId, goalId, stageId)` → Stage (+ пересчёт прогресса цели)

### Шаг 2.6 — GoalController (45 мин)
**Изменить:** `service/JwtService.java` — добавить `extractUserId(token)` с claim `user_id`
**Изменить:** `filter/JwtAuthFilter.java` — передавать userId в Authentication principal
**Создать:** `controller/GoalController.java` (`@RequestMapping("/api/goals")`):
- `POST /` — создать цель (201)
- `GET /?page=1&size=20` — список целей с пагинацией
- `GET /{id}` — детали цели со stages
- `POST /{id}/stages` — добавить этап (201)
- `PUT /{id}/stages/{stageId}/complete` — завершить этап
- `POST /{id}/ai-decompose` — заглушка (202, "processing")

### Шаг 2.7 — Полное E2E тестирование (45 мин)
Postman-сценарий:
1. Register → Confirm email → Login
2. Создать цель → Список целей → Детали цели
3. Добавить этап → Завершить этап → Проверить прогресс
4. AI decompose → 202
5. Refresh token
6. Проверить изоляцию данных между пользователями

### Шаг 2.8 — Финальная чистка (30 мин)
Убрать тестовый код, проверить все эндпоинты, обновить Swagger-документацию
